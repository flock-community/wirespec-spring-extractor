# Gradle plugin for `wirespec-spring-extractor`

**Status:** Design approved 2026-05-12.
**Scope:** Add a Gradle plugin that exposes the same extraction capability as the
existing Maven plugin, and add two integration tests that drive the published
Gradle plugin against fixture Gradle projects (Kotlin + Java/Spring).

## Goal

Consumers using Gradle get the same one-line install / zero-config wiring as
Maven consumers. Applying the plugin and running `gradle build` produces `.ws`
files under `build/wirespec/`, matching the Maven plugin's output under
`target/wirespec/`.

The Gradle plugin is a thin shim over `WirespecExtractor.extract(ExtractConfig)`
in `:extractor-core`, exactly as `:extractor-maven-plugin` is.

## Non-goals

- Multi-platform / Kotlin Multiplatform support — JVM only, like the Maven plugin.
- Groovy DSL fixtures or DSL-flavour testing. Both fixtures use the Kotlin DSL.
- Sharing fixture sources / verifier code between the Maven and Gradle IT modules.
  Initial duplication is accepted; can be revisited if maintenance hurts.

## Distribution & coordinates

- **Plugin ID:** `community.flock.wirespec.spring.extractor`
- **Artifact group:** `community.flock.wirespec.spring` (same as Maven plugin)
- **Artifact ID:** `wirespec-spring-extractor-gradle-plugin`
- **Plugin marker ID:** `community.flock.wirespec.spring.extractor` (auto-generated by `java-gradle-plugin`)
- **Publish targets:**
  - Gradle Plugin Portal (via `com.gradle.plugin-publish`) — primary distribution
    so consumers can use `plugins { id("…") version "…" }`.
  - Build-local `it-repo` (via `maven-publish`) — used by integration tests.
- **Minimum Gradle version:** `>= 8.5` (parallels the Maven plugin's
  `requiredMavenVersion=3.9.0` choice).

## Core API change (one rename)

The Gradle JVM plugin produces classes in **multiple directories** by default
(`build/classes/kotlin/main/`, `build/classes/java/main/`). The Maven plugin
only ever has one (`target/classes`). The current `ExtractConfig` reflects the
Maven view:

```kotlin
data class ExtractConfig(
    val classesDirectory: File,             // singular
    val runtimeClasspath: List<File>,
    val outputDirectory: File,
    val basePackage: String? = null,
    val log: ExtractLog = ExtractLog.NoOp,
)
```

This is renamed to:

```kotlin
data class ExtractConfig(
    val classesDirectories: List<File>,     // plural
    val runtimeClasspath: List<File>,
    val outputDirectory: File,
    val basePackage: String? = null,
    val log: ExtractLog = ExtractLog.NoOp,
)
```

The public API was added in commit `37ac623 feat(core): add WirespecExtractor
public API`; nothing has been released. Treat as pre-1.0 and rename cleanly
rather than carrying both fields with deprecation.

Touch list:

1. `extractor-core/ExtractConfig.kt` — rename field.
2. `extractor-core/WirespecExtractor.kt` — the "no classes" guard checks each
   directory; the URL collection adds each to the classloader; controller
   scanning sees all of them via the classloader.
3. `extractor-core/classpath/ClasspathBuilder.kt` — `collectUrls` accepts a list
   of class output directories (or the call site iterates).
4. `extractor-maven-plugin/ExtractMojo.kt` — pass
   `classesDirectories = listOf(File(project.build.outputDirectory))`.
5. `extractor-gradle-plugin/ExtractWirespecTask.kt` — pass
   `classesDirectories = classesDirs.files.toList()`.
6. Tests that construct `ExtractConfig` directly: mechanical update.

**Alternative considered & rejected**: keep `classesDirectory: File` and put the
extra Gradle output dir on `runtimeClasspath`. This works for scanning but
breaks the existence-check guard ("did `compile` run?"), since that guard only
inspects `classesDirectory`. With Kotlin-only or Java-only modules the wrong
dir on that field yields a false-negative error.

## Gradle plugin — API surface

### Plugin entry — `WirespecExtractorPlugin.kt`

```kotlin
class WirespecExtractorPlugin : Plugin<Project> {
    override fun apply(project: Project) {
        val ext = project.extensions.create("wirespec", WirespecExtractorExtension::class.java).apply {
            outputDir.convention(project.layout.buildDirectory.dir("wirespec"))
            // basePackage left without a convention → null/unset means "scan everything"
        }

        // Wait for the JVM plugin. Kotlin-JVM applies JavaPlugin under the hood.
        project.plugins.withType(JavaPlugin::class.java) {
            val main = project.extensions.getByType(SourceSetContainer::class.java).getByName("main")

            val extractTask = project.tasks.register("extractWirespec", ExtractWirespecTask::class.java) { t ->
                t.group = "wirespec"
                t.description = "Extract Wirespec .ws files from Spring controllers."
                t.classesDirs.from(main.output.classesDirs)
                t.runtimeClasspath.from(main.runtimeClasspath)
                t.outputDirectory.convention(ext.outputDir)
                t.basePackage.convention(ext.basePackage)
                t.dependsOn(main.output.classesDirs)   // ensures compile runs first
            }

            // Auto-wire: every `gradle build` / `gradle assemble` runs the extractor.
            project.tasks.named("assemble") { it.dependsOn(extractTask) }
        }
    }
}
```

### Extension — `WirespecExtractorExtension.kt`

```kotlin
abstract class WirespecExtractorExtension {
    abstract val outputDir: DirectoryProperty
    abstract val basePackage: Property<String>
}
```

### Task — `ExtractWirespecTask.kt`

```kotlin
abstract class ExtractWirespecTask : DefaultTask() {
    @get:InputFiles @get:SkipWhenEmpty
    abstract val classesDirs: ConfigurableFileCollection

    @get:Classpath
    abstract val runtimeClasspath: ConfigurableFileCollection

    @get:OutputDirectory
    abstract val outputDirectory: DirectoryProperty

    @get:Input @get:Optional
    abstract val basePackage: Property<String>

    @TaskAction
    fun run() {
        try {
            WirespecExtractor.extract(
                ExtractConfig(
                    classesDirectories = classesDirs.files.toList(),
                    runtimeClasspath   = runtimeClasspath.files.toList(),
                    outputDirectory    = outputDirectory.asFile.get(),
                    basePackage        = basePackage.orNull,
                    log                = GradleExtractLog(logger),
                )
            )
        } catch (e: WirespecExtractorException) {
            throw GradleException(e.message ?: "Wirespec extraction failed", e)
        }
    }
}
```

### Log adapter — `GradleExtractLog.kt`

```kotlin
internal class GradleExtractLog(private val logger: org.gradle.api.logging.Logger) : ExtractLog {
    override fun info(msg: String) { logger.info(msg) }   // lifecycle-level visibility is fine; matches Maven
    override fun warn(msg: String) { logger.warn(msg) }
}
```

### Cache correctness

`@Classpath` on `runtimeClasspath`, `@InputFiles` on `classesDirs`, and
`@OutputDirectory` on the output dir make the task fully up-to-date and build-
cache friendly. The task reruns only when controllers or runtime dependencies
actually change.

### Consumer usage

```kotlin
plugins {
    kotlin("jvm") version "2.1.20"
    id("community.flock.wirespec.spring.extractor") version "0.1.0"
}

wirespec {
    basePackage.set("com.acme.api")
    // outputDir defaults to build/wirespec — override if needed
}
```

`gradle build` (or `gradle assemble`) auto-runs `extractWirespec`.
`gradle extractWirespec` runs it directly.

## Integration tests — `:integration-tests-gradle`

### Module `build.gradle.kts`

```kotlin
plugins { alias(libs.plugins.kotlin.jvm) }

description = "End-to-end tests that drive the Gradle plugin against fixture Gradle/Spring/Kotlin projects via TestKit."

kotlin { jvmToolchain(21) }

dependencies {
    testImplementation(libs.junit.jupiter)
    testImplementation(libs.kotest.assertions)
    testImplementation(gradleTestKit())
}

val itRepoDir = rootProject.layout.buildDirectory.dir("it-repo")

tasks.test {
    useJUnitPlatform()
    // Same it-repo the Maven IT uses — both plugins resolve from one place.
    dependsOn(":extractor-gradle-plugin:publishAllPublicationsToItLocalRepository")
    dependsOn(":extractor-core:publishMavenPublicationToItLocalRepository")

    systemProperty("it.pluginVersion", project.version.toString())
    systemProperty("it.repo",          itRepoDir.get().asFile.absolutePath)
    systemProperty("it.fixturesRoot",  layout.projectDirectory.dir("src/it").asFile.absolutePath)
    systemProperty("it.workRoot",      layout.buildDirectory.dir("it-work").get().asFile.absolutePath)
    systemProperty("it.testKitDir",    layout.buildDirectory.dir("it-testkit").get().asFile.absolutePath)

    testLogging { showStandardStreams = false; events("failed") }
}
```

`publishAllPublicationsToItLocalRepository` (vs. the Maven plugin's
`publishMavenPublicationToItLocalRepository`) publishes both the jar publication
and the Gradle **plugin marker** artifact. The marker is what makes
`plugins { id("…") version "…" }` resolve from the IT repo.

### Test runner — `GradleFixtureBuildTest.kt`

Mirrors `FixtureBuildTest.kt`:

```kotlin
class GradleFixtureBuildTest {
    private val pluginVersion = sysProp("it.pluginVersion")
    private val itRepo        = File(sysProp("it.repo"))
    private val fixturesRoot  = File(sysProp("it.fixturesRoot"))
    private val workRoot      = File(sysProp("it.workRoot"))
    private val testKitDir    = File(sysProp("it.testKitDir"))

    @TestFactory
    fun `each fixture builds and produces expected wirespec output`(): List<DynamicTest> =
        fixturesRoot.listFiles { f -> f.isDirectory && File(f, "settings.gradle.kts").isFile }!!
            .sortedBy { it.name }
            .map { fixture -> DynamicTest.dynamicTest(fixture.name) { runFixture(fixture) } }

    private fun runFixture(fixture: File) {
        val workDir = File(workRoot, fixture.name).also { it.deleteRecursively(); it.mkdirs() }
        copyFixture(fixture.toPath(), workDir.toPath())
        substituteTokens(workDir, mapOf(
            "@project.version@" to pluginVersion,
            "@itRepo@"          to itRepo.toURI().toString(),
        ))

        GradleRunner.create()
            .withProjectDir(workDir)
            .withTestKitDir(testKitDir)            // isolated; never touches user's ~/.gradle
            .withArguments("assemble", "--stacktrace", "--no-daemon")
            .forwardOutput()
            .build()

        when (fixture.name) {
            "basic-kotlin-app" -> verifyBasicKotlinApp(workDir)
            "basic-spring-app" -> verifyBasicSpringApp(workDir)
            else -> error("No verifier registered for fixture ${fixture.name}")
        }
    }

    // verifyBasicKotlinApp / verifyBasicSpringApp: copy of the Maven IT verifiers,
    // with output path "target/wirespec" → "build/wirespec".
}
```

### Two integration tests

The `@TestFactory` emits one dynamic test per fixture:

1. **`basic-kotlin-app`** — Kotlin DSL build, Kotlin sources. Verifies same
   assertions as the Maven Kotlin IT: controller/types content, JDK type
   filtering, Kotlin nullability, backticked field names, no coroutine-internal
   leaks.
2. **`basic-spring-app`** — Kotlin DSL build, Java sources. Verifies same
   assertions as the Maven Java IT: controller/types content, JDK type
   filtering, backticked field names.

### Why TestKit (vs. shelling out to `gradlew`)

TestKit runs Gradle in-process against the cross-version provider, isolates the
Gradle user home via `withTestKitDir`, and gives structured `BuildResult`
access. No need for the consumer's machine to have `gradle` on `PATH`. The
Maven IT shells out only because Maven has no in-process equivalent.

### Risks / mitigations

- **Plugin marker resolution.** Handled by publishing the marker into
  `it-repo` via `publishAllPublicationsToItLocalRepository`.
- **Gradle user home pollution.** Handled by `withTestKitDir(testKitDir)` —
  points to `build/it-testkit/`, never touches `~/.gradle`.
- **Daemon test pollution.** `--no-daemon`. `clean` is implicit because
  `workDir` is recreated.
- **Slow first run** (TestKit downloads the Gradle distribution): one-time,
  cached in `testKitDir`.

## Out of scope (deferred)

- Sharing fixture sources between `:integration-tests` and
  `:integration-tests-gradle`. Initial duplication is accepted.
- Sharing verifier assertion code between the two IT modules.
- Disable-auto-wire flag (`wirespec { autoBind.set(false) }`). The Maven plugin
  has no equivalent and no consumer has asked.
- Groovy-DSL fixture. Both Gradle fixtures use Kotlin DSL.
- KMP / non-JVM support.

## README impact

The repository README currently documents only the Maven plugin. After this
change it should also document the Gradle plugin: a `plugins { id("…") version
"…" }` block, the `wirespec { … }` extension, and the auto-wire behaviour.
Wording can be added as part of implementation (not a design decision).
